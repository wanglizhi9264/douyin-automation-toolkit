# ADR 0003：作品归一为媒体分段并逐段保存检查点

- 状态：Accepted
- 日期：2026-07-16

## 背景

抖音作品不只有单个 MP4。图文包含有序原图，live photo 可能同时含图片和视频，部分作品由接口明确返回多个视频段。只保存一个 `videoUrl` 会丢内容；只在整条作品结束时写状态，会在中断后重复下载已经完成的分段。

## 决策

1. 所有列表项归一为有序 `mediaParts`，以 `partId` 标识图片或视频段，并从分段数量计算 `mediaType`。
2. 普通单视频保留 `<awemeId>.mp4` 路径；多段视频和图片使用 `<awemeId>/<序号>` 目录。
3. 每个视频段独立使用现有最高画质候选排序和最多四候选回退，不因支持多媒体改变画质策略。
4. 每个分段成功后立即写 `downloadedMediaParts[partId]` 和文件夹检查点；只有全部内容段成功后作品才是 `downloaded`。
5. 插件和 backend 共用媒体解析及记录字段，同一个授权目录可以在两端继续。
6. 图文存在时不重复下载顶层合成播放视频；嵌套 live-photo 视频属于独立内容段。

## 结果

- 暂停、刷新、进程退出和单段失败后可从未完成段续传。
- manifest 能完整表达一个作品的所有内容文件。
- 新接口结构只有在明确暴露媒体地址并补充 fixture 后才能加入解析集合。
- 老的单视频路径与已下载记录继续兼容；记录 schema 新增字段但不删除旧字段。

## 验证

- `plugin/scripts/test-media-parts.mjs` 覆盖多图、live photo、显式多视频和续传静态契约。
- 插件记录恢复测试覆盖部分完成作品。
- backend 核心测试覆盖分段路径和最高画质候选顺序。
- 真实案例按测试发布手册人工冒烟。
